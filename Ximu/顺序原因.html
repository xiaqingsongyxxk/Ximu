1 初始化项目结构 2026-04-02 第 0 层 · 基础设施

📁 创建的文件

backend/pyproject.toml # Python 项目依赖配置
backend/.python-version # Python 3.13 版本锁定
backend/shared/__init__.py
backend/shared/types/__init__.py
backend/shared/types/strict_model.py # 严格基础模型 (extra="forbid", camelCase)
backend/shared/types/resume.py # 简历类型定义 (ResumeSchema + 所有板块类型)

❓ 为什么先创建这些？

pyproject.toml
所有 Python 项目的起点。没有它就无法安装依赖、定义工具链、运行任何代码
.python-version
锁定 Python 3.13，确保所有开发者和 CI 环境使用一致的版本，避免兼容性问题
strict_model.py
整个类型系统的根基。所有业务类型都继承自 StrictBaseModel，它定义了禁止额外字段、自动 camelCase 别名等核心行为。没有它，后续任何数据模型都无法定义
resume.py
业务核心类型。简历系统的所有数据结构（ResumeSchema、PersonalInfo、WorkExperience 等 11 种板块类型）都从这里开始

🔗 继承链

StrictBaseModel ← ResumeSchema ← PersonalInfoSection ← WorkExperienceSection ← ...其他 9 种板块
→ strict_model.py 和 resume.py 是整个类型系统的 根节点，所有后续模块都直接或间接依赖它们
2 Parser 模块 — PDF 简历解析（MVP） 2026-04-03 第 1 层 · 核心功能

📁 创建的文件

apps/parser/
├── __init__.py
├── router.py # API 路由 (POST /parser, POST /parser/retry)
├── schemas.py # 请求/响应模型 (ParserResult)
├── pdf_parser.py # PDF 文本提取 (pymupdf)
├── prompt.py # LLM 解析提示词
├── call_llm.py # LLM 调用封装
├── storage.py # 上传文件存储
└── service.py # 业务逻辑 (编排解析流程)

❓ 为什么最先实现 parser？

核心功能
用户的第一个需求是"上传简历 PDF，提取内容"。没有解析功能，其他所有功能都无从谈起
独立性最强
parser 不依赖其他业务模块，只需要最基础的类型定义。这降低了初始开发的复杂度
MVP 思维
先做最小可用产品（Minimum Viable Product），验证"PDF → 结构化数据"这条核心链路是否可行
降低风险
如果 PDF 解析这个关键技术难点无法攻克，整个项目就没必要继续。先做最有风险的部分

🔗 依赖关系

shared/types/strict_model.py shared/types/resume.py pymupdf openai / anthropic

📐 技术流程

用户上传 PDF
    ↓
pdf_parser.py ← pymupdf 提取文本
    ↓
call_llm.py + prompt.py ← LLM 解析为非结构化 → 结构化数据
    ↓
storage.py + service.py ← 创建 Resume + ResumeSection 记录
    ↓
router.py → 返回 task_id 给前端轮询结果
3 Resume Assistant — AI 对话优化简历 2026-04-05 第 1 层 · 差异化功能

📁 创建的文件

apps/resume_assistant/
├── __init__.py
├── router.py # SSE 流式对话 API
├── schemas.py # ChatRequest / ChatResponse
├── conversation_store.py # 对话历史内存存储
├── agent_service.py # Agent 核心服务
├── task_service.py # 后台任务管理
├── prompt.py # 系统提示词
├── agent/ # Agent 核心 (core, runtime, state, context...)
│ └── core.py, runtime.py, events.py, state.py, ...
└── tools/ # Agent 工具 (section_info, update_section, ...)

❓ 为什么第二实现 AI 助手？

差异化功能
简历工具很多（几乎每个平台都有解析功能），但"AI 对话优化简历"是真正的竞争亮点。先做核心，再做差异
可利用 parser
AI 助手需要理解简历结构才能进行优化。parser 已经定义了完整的简历类型体系，助手可以直接复用
技术复杂度高
需要 Agent 工具调用循环、SSE 流式推送、对话状态管理——是技术难度最高的模块，尽早攻克为后续功能铺路
用户价值高
直接解决了"不知道怎么优化简历"的问题。用户上传简历后，可以立即通过对话获得改进建议

🔗 依赖关系

shared/types/strict_model.py shared/types/resume.py shared/types/base_tool.py shared/types/messages.py openai / anthropic
📌 注意：这个阶段 resume_assistant 的 LLM 调用是各自独立的（有自己的 call_llm 逻辑）。 此时 shared/api/ 客户端尚未提取——这是在阶段 4 重构时才统一的。
4 Shared 模块提取 — 架构优化 2026-04-07 第 2 层 · 架构重构

📁 创建/重构的文件

shared/
├── database.py # 异步 SQLAlchemy 引擎 + get_session
├── models.py # ORM 模型 (Resume, ResumeSection, UserConfig, ...)
├── api/
│ ├── client.py # 统一 LLM 客户端接口
│ ├── openai_client.py # OpenAI 兼容实现
│ ├── errors.py # LLM 错误类型
│ └── usage.py # Token 用量跟踪
├── exceptions/
│ └── base.py # APIException 基类
└── types/
├── messages.py # 对话消息类型
├── jd_analysis.py # JD 分析类型
├── base_tool.py # Agent 工具类型
└── mixins.py # API Response Mixin

❓ 为什么这时候提取 shared？

发现重复代码
parser 有自己的 call_llm.py，resume_assistant 也有自己的 LLM 调用逻辑——两套几乎相同的实现
避免重复累积
即将开发的 jd_analysis、cover_letter 也都需要调用 LLM。如果现在不统一，后续会重复更多
依赖关系清晰
database.py 和 models.py 被所有业务模块依赖，放到 shared 中让依赖方向明确：业务 → shared
重构最佳时机
核心功能已跑通、新增功能尚未开始——功能跑通后立即重构，避免技术债累积

🔄 重构内容

重构前
parser/call_llm.py ← 自己的 LLM 调用
resume_assistant/ ← 自己的 LLM 调用
重构后
shared/api/client.py ← 统一 LLM 客户端
shared/api/openai_client.py ← OpenAI 兼容实现
parser → 调用 shared/api/
resume_assistant → 调用 shared/api/

🔗 依赖关系

5 Config 模块 — AI 提供商配置管理 2026-04-09 第 2 层 · 基础设施

📁 创建的文件

apps/config/
├── __init__.py
├── schemas.py # ProviderConfig, ProviderConfigUpdate, ProviderSwitch
└── router.py # GET/PUT /config/provider, PATCH /config/provider/switch

❓ 为什么这时候添加配置模块？

需求驱动
parser 和 resume_assistant 目前都在硬编码 API Key。用户不可能接受固定某个 API Key
用户需要配置
用户需要能自由切换不同的 AI 提供商（OpenAI / Anthropic）、配置不同的模型和 API Key
简单但重要
本质上是 key-value 存储（UserConfig 表），没有复杂业务逻辑，实现成本低但收益大
被依赖方
所有需要调用 LLM 的模块都依赖配置信息。config 是"被依赖"的模块，应该先于"依赖它的模块"完成

🔗 依赖关系

shared/database.py shared/models.py (UserConfig)
✅ config 不依赖任何其他 apps 模块 → 可以独立开发、独立测试

├── __init__.py

❓ 为什么这时候添加简历管理？

数据需要管理
parser 解析出的简历数据需要展示、编辑、组织。没有 resume 模块，用户无法查看和管理自己的简历
Workspace 概念
引入了"主简历（workspace）+ 子简历（按 JD 优化）"的概念。这是整个产品的核心数据模型
自然演进
parser → 有数据 → 需要管理数据。这是一个自然的产品演进路径
JD 分析的前提
JD 匹配分析需要针对具体的简历和职位描述。没有简历管理功能，jd_analysis 无法实现

📐 Workspace 架构

Workspace（主简历）
├── 子简历 1（针对 JD A）
├── 子简历 2（针对 JD B）
└── 子简历 3（针对 JD C）

# 每个子简历有独立的板块内容
# 但共享主简历的模板和主题
# 创建子简历时，从主简历复制所有板块（copy_sections_from_workspace）

7 JD Analysis — 职位匹配分析 2026-04-11 第 3 层 · 业务完善

❓ 为什么和 resume 同时添加？

独立功能
可以单独测试和使用，不阻塞其他功能的开发

├── ATS 兼容性评分（0-100）
├── 关键词匹配列表
├── 缺失关键词列表
└── 各板块优化建议

9 Work 任务系统 — 统一异步任务管理 2026-04-16 第 2 层 · 基础设施重构

📁 创建的文件

apps/work/
├── __init__.py
├── router.py # 任务 CRUD API
└── sse.py # SSE 流式任务状态推送

└── task_state.py # 内存任务状态管理
shared/types/
├── work.py # WorkTaskSchema
└── task.py # TaskStatus / TaskType 枚举

❓ 为什么需要任务系统？

异步任务
JD 分析、求职信生成、PDF 导出都是耗时操作（几秒到十几秒），不能同步等待
状态追踪
用户需要知道任务进度：pending → running → success/error。没有任务系统，前端无法得知状态
SSE 推送
实时将任务状态变化推送给前端，而不是让前端轮询。提供更好的用户体验
统一管理
parser、jd_analysis、cover_letter、export 都需要异步任务管理。统一设计避免各自为政

📐 任务状态机

pending → running → success
             ↓
          error

10 Export — PDF 导出 2026-04-23 第 3 层 · 输出闭环

📁 创建的文件

├── __init__.py
├── router.py # POST /export/pdf
├── browser_manager.py # Playwright 浏览器管理
├── pdf_generator.py # HTML → PDF 渲染
└── service.py # 导出业务逻辑

❓ 为什么快最后才添加导出？

用户价值高
简历最终需要导出为 PDF/HTML/TXT/JSON，但这是"输出"环节——前提是内容已经完善
技术复杂
需要 Playwright 浏览器管理、HTML 渲染为 PDF、浏览器生命周期管理。引入了一个复杂的系统依赖
依赖其他模块
需要完整的简历数据（resume + resume_section）才能导出。这些数据在此之前才就绪
核心功能优先
在导出之前，必须先确保"导出什么"是正确的——即解析、优化、分析的功能已经完善

📐 技术流程

HTML 模板（前端渲染）
    ↓
service.py 请求后端导出
    ↓
browser_manager.py 获取/启动 Playwright Chromium
    ↓
pdf_generator.py 渲染 HTML 为 PDF
    ↓
返回 PDF 文件给前端下载

❓ 为什么最后组装？

模块化开发

◆ 核心构建理念

MVP 思维

先做核心功能（PDF 解析 - parser），再完善周边（求职信、模板管理）。每个阶段都能交付可用的产品。

依赖驱动

先实现被依赖的模块（shared/types → shared/database → shared/api → config → 业务 apps），再实现依赖别人的。

重构时机

功能跑通后立即重构（phase 4 提取 shared）。避免技术债累积，越早重构成本越低。

用户价值

高价值功能优先（parser → resume_assistant → jd_analysis），低价值功能靠后（template）。
